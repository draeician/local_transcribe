### Root Cause Analysis: YouTube 2026 "SABR" Throttling & 403 Forbidden Errors

#### The Issue

In early 2026, YouTube escalated its "SABR" (Server-Side Ad-Insertion and Rendering) protocol. This update introduced a strict **"n-challenge"**—a rotating JavaScript-based signature required to authorize media downloads.

* **Symptoms:** Automated tools like `yt-dlp` and `local_transcribe` began hitting `HTTP 403: Forbidden` errors during fragment downloads.
* **The "Blindness" Effect:** Without solving the JavaScript challenge, YouTube would only serve low-bitrate, fragmented HLS (m3u8) streams. Standard high-quality direct audio formats (like `itag 140`) disappeared from the available formats list.

---

#### What Was Tried (And Why It Failed)

1. **Updating `yt-dlp` and Cookies:**
* **Result:** Failure.
* **Why:** Modern "n-challenge" puzzles are session-specific and dynamic. Even with valid cookies, if the client cannot execute the JavaScript puzzle to generate the correct signature, YouTube rejects the request.


2. **Using Android/iOS Player Clients:**
* **Result:** Failure (PO Token required).
* **Why:** YouTube began requiring **GVS PO Tokens** (Proof of Origin) for mobile API clients. These tokens are generated by real hardware devices and are extremely difficult to spoof in a Linux environment without specific browser-based generator plugins.


3. **Local Node.js Runtime:**
* **Result:** Intermittent Failure / Silent Crashes.
* **Why:** While Node.js was installed on the system, the `pipx` isolated virtual environment used by `local_transcribe` had difficulty reliably calling the local Node binary across the environment boundary.


4. **`curl-cffi` Impersonation:**
* **Result:** Silent Error (Empty `[warn]` messages).
* **Why:** While `curl-cffi` successfully spoofs a Chrome TLS fingerprint, it introduced instability within the Python subprocess during the initial `yt-dlp` handshake, causing the strategy to fail before it could even attempt a download.



---

#### The Fix: The "Harmony 2026" Stable Strategy

The final solution relied on a combination of environment bridging and forcing the "Web" player client, which is currently the only client that prioritizes JavaScript execution over hardware PO Tokens.

**1. The JavaScript Solver (Deno):**
We installed **Deno** (referred to in community lore as "Dino") and created a global symlink in `/usr/local/bin/deno`. This ensured that the `pipx` virtual environment could "see" a high-performance JS runtime regardless of its path isolation. This solved the "n-challenge," revealing the high-quality **Format 140 (m4a)** audio stream.

**2. The Web-Client Strategy:**
We updated `src/local_transcribe/services/downloader.py` to move away from mobile clients (which require PO Tokens) and forced the **`web`** client. This client triggers the signature logic that Deno is capable of solving.

**3. Backend Stabilization:**
We reverted the default networking backend to `requests` for the primary strategy while keeping `curl_cffi` as an optional injection. This eliminated the silent crashes seen during the impersonation attempts.

---

#### Final Configuration Summary

* **Runtime Dependency:** Deno linked to `/usr/local/bin`.
* **Strategy Order:** 1.  `stable-web`: Uses the standard Web client headers + local Deno solver.
2.  `ios-fallback`: Standard fallback for restricted videos.
* **Result:** Success. The system now correctly identifies the direct m4a audio stream, bypasses the fragmented HLS paths, and avoids 403 errors.
